# 基于机制本体的个性化 AI 建议设计

## 目标

在现有 AI 教练提示词中加入敌方英雄的机制特性和针对方式，并根据玩家当前英雄、1–5 号位、已有装备及游戏阶段筛选出玩家本人能够执行的建议。

本功能以 `Dota2MechanismOntology` 的结构化数据为事实来源，但不把完整本体直接发送给模型。程序先在本地检索和压缩知识，再把有限、明确的机制参考加入现有用户消息。

成功标准：

- 敌方阵容被识别后，提示词包含每名敌方英雄最重要的机制特性。
- 针对建议与玩家位置和英雄职责相符。
- 已购买装备不会继续作为新的出装建议重复推荐。
- 高优先级团队义务可以突破位置限制，例如敌方有隐身能力时提醒辅助准备反隐。
- 本体缺失、数据损坏或英雄无法匹配时，现有 AI 教练仍可正常工作。
- 源码运行和 PyInstaller 单文件运行均能加载本体资源。

## 当前基础

现有 `StateExtractor` 已负责：

- 从 minimap 累积双方阵容。
- 将英雄内部名称转换为中文名。
- 提取玩家当前英雄、装备、技能、位置和局势。

现有 `PromptBuilder` 在固定前缀中只输出双方阵容名称，尚未使用机制知识。

本体目录提供：

- `hero_profiles.json`：127 名英雄的派生特征、物品需求、受克制类别和技能升级信息。
- `relation_edges.json`：英雄、物品和特征之间的 `counter`、`depend`、`enable`、`dispel` 关系，包含强度、阶段和证据。
- `retrieval_blocks.json`：已经压缩的英雄画像和其他检索块。
- `indexes.json`：英雄、物品、关系和特征索引。
- `item_profiles.json` 与 `skills.json`：物品和技能的详细机制证据。

第一版只依赖 `hero_profiles.json`、`relation_edges.json` 和 `indexes.json`。其余文件保留为后续精细化能力，不在第一版扩大范围。

## 方案选择

采用“本体候选检索 + 本地软评分 + 提示词注入”。

不采用以下两种方案：

1. 直接注入完整英雄检索块：实现简单，但五名敌人的文本量和噪声较大，模型容易机械照抄。
2. 严格按位置过滤：输出干净，但本体标签不完整时容易漏掉关键反制，例如隐身、强回复或必须准备的团队道具。

软评分方案既保留位置适配，又允许关键反制突破常规职责限制。

## 架构

新增 `src/advisor/ontology.py`，内部包含两个职责清晰的组件。

### `OntologyRepository`

负责一次性加载和索引本体数据，不理解游戏局势。

主要接口：

```python
class OntologyRepository:
    def get_hero_profile(self, hero_name: str) -> Optional[dict]:
        ...

    def get_counter_edges(self, hero_name: str) -> list[dict]:
        ...
```

行为：

- 使用 `resource_path()` 解析源码和 PyInstaller 环境中的路径。
- 加载时验证顶层 JSON 类型和必要字段。
- 根据 `indexes.json` 快速定位英雄画像。
- 根据 `edges_by_target` 或加载时建立的内存索引查询指向敌方英雄的 `counter` 关系。
- 数据只加载一次，整局查询复用内存结果。
- 加载失败后记录一次警告并进入不可用状态，不在每次建议时重复报错。

### `EnemyKnowledgeBuilder`

负责把本体候选转换为当前玩家可执行的机制参考。

主要接口：

```python
class EnemyKnowledgeBuilder:
    def build(
        self,
        enemy_heroes: list[str],
        player_hero: str,
        role: str,
        owned_items: set[str],
        game_time: float,
    ) -> str:
        ...
```

输出是紧凑文本，不返回模型最终建议。例如：

```text
【敌方机制参考】
- 幻影刺客：高爆发、高机动、被动依赖。优先保留控制防切入；物理核心可考虑金箍棒或破坏效果。
- 力丸：隐身、爆发。团队必须准备粉和真眼；你是4号位，优先承担反隐。
```

该组件不调用网络 API，也不修改 GSI 状态，便于独立测试。

### `PromptBuilder`

`PromptBuilder` 继续只承担提示词组装，但接收 `EnemyKnowledgeBuilder` 作为可选依赖。

机制参考放在双方阵容之后、动态局势之前。阵容未收集完整时，可以为已知敌方英雄生成部分知识；随着阵容补全，固定前缀重新生成并缓存完整结果。

提示词增加约束：

- 本体内容是机制参考，不是必须执行的固定答案。
- 最终建议必须结合当前血量、位置、装备、阶段和视野。
- 不要推荐玩家已经拥有的装备。
- 出装建议应符合玩家英雄与分路职责，除非属于必须由团队承担的关键反制。

## 敌方识别

玩家阵营来自 `player.team_name`：

- 玩家为天辉时，夜魇阵容是敌方。
- 玩家为夜魇时，天辉阵容是敌方。

当前阵容列表使用中文英雄名，能够直接匹配 `hero_profile_by_name`。若中文名无法匹配，则跳过该英雄并保留其他敌人的知识，不中断整次构建。

## 玩家上下文

用于评分的输入包括：

- 玩家英雄中文名。
- 玩家分路 `1`–`5`。
- 当前物品内部名称和可映射的中文物品名。
- 游戏时间对应的阶段。

阶段定义：

- `0–8` 分钟：对线期。
- `8–15` 分钟：前期。
- `15–30` 分钟：中期。
- `30` 分钟以后：后期。

本体关系中的阶段映射：

- `all`：所有阶段有效。
- `midgame`：中期及以后优先。
- `lategame`：后期优先。
- `teamfight`：中期及以后提高权重，但不作为绝对过滤条件。

## 软评分规则

每个指向敌方英雄的 `counter` 关系生成一个候选。候选基础分为关系的 `strength`。

第一版采用可读、确定性的加减分规则：

| 条件 | 分数变化 |
|---|---:|
| 关系阶段包含 `all` | +5 |
| 当前阶段与关系阶段匹配 | +10 |
| 候选装备符合玩家位置职责 | +15 |
| 候选属于玩家英雄画像中的物品需求类别 | +10 |
| 候选是关键团队义务 | +20 |
| 玩家已经拥有该物品 | 排除 |
| 与玩家位置明显冲突且不是团队义务 | -20 |
| `strength < 70` | 默认排除 |

位置职责映射采用有限的装备特征集合：

- 1号位：进攻装、魔免装、留人装、破坏装、续航装。
- 2号位：进攻装、位移装、留人装、魔免装、技能强化装。
- 3号位：保命装、团队装、位移装、留人装、反制装。
- 4号位：位移装、保命装、反隐装、视野装、留人装、团队装。
- 5号位：保命装、反隐装、视野装、团队装、驱散装。

关键团队义务第一版包括：

- 敌方具有隐身特征：反隐装、视野装。
- 敌方具有高回复特征：禁疗装。
- 敌方具有幻象核心特征：具备范围清理能力的候选。

若本体暂时不能给出具体且可信的物品，输出打法类别而不是编造物品名称。

## 压缩与去重

为控制提示词长度：

- 每名敌方英雄最多保留 3 个最高分特性。
- 每名敌方英雄最多保留 2 条个人可执行针对方式。
- 同一物品或同一策略只出现一次。
- 多名敌人共享同一威胁时合并为团队级提示，例如“敌方两名英雄具有隐身能力”。
- 最多输出 5 名敌人的摘要和 2 条团队级提示。
- 整个机制参考设定字符上限；超过时优先保留高强度关系、团队义务和与玩家位置最匹配的内容。

排序优先级：

1. 关键团队义务。
2. 评分更高的候选。
3. 当前阶段匹配项。
4. 敌方阵容中的原始顺序，保证输出稳定。

## 配置

在 `ai_advisor` 下新增：

```yaml
ontology:
  enabled: true
  path: "./Dota2MechanismOntology"
  min_counter_strength: 70
  max_traits_per_hero: 3
  max_counters_per_hero: 2
  max_context_chars: 1800
```

所有字段提供默认值。旧配置不包含 `ontology` 时默认启用仓库内本体；若资源不存在则自动降级。

## 数据流

1. `StateExtractor` 持续收集双方阵容。
2. `AiAdvisor.update()` 达到触发条件。
3. `PromptBuilder` 判断敌方阵容。
4. `EnemyKnowledgeBuilder` 查询敌方画像和反制关系。
5. 候选结合玩家英雄、位置、装备和阶段进行软评分。
6. 结果压缩为“敌方机制参考”。
7. `PromptBuilder` 将参考信息加入用户消息。
8. 模型结合实时 GSI 状态输出原有 `analysis`、`command`、`item` JSON。

不改变触发器、网络客户端、后台线程或事件输出结构。

## 资源打包

更新 `Dota2GSI.spec` 和 `build.bat`，将 `Dota2MechanismOntology` 作为数据目录打入可执行文件。

运行时路径统一通过：

```python
resource_path("Dota2MechanismOntology")
```

不依赖当前工作目录。

## 错误处理

- 本体目录不存在：打印一次警告，返回空机制参考。
- JSON 解析失败：打印一次包含文件名的警告，禁用本体功能。
- 单个英雄不存在：跳过该英雄。
- 单条关系缺少字段：跳过该关系。
- 评分后无合格候选：仍可输出英雄特性；若画像也缺失则不输出该英雄。
- 文本超过限制：按排序结果截断完整条目，不截断到半句话。

所有降级都不得阻止 AI API 调用。

## 测试

新增本体组件测试，使用临时目录中的最小 JSON fixture，避免测试依赖完整大数据文件。

覆盖：

- 正确加载英雄画像和目标关系索引。
- 玩家为天辉或夜魇时选择正确敌方阵容。
- 1号位和5号位面对同一敌人得到不同优先级建议。
- 关键反隐义务能够突破普通位置过滤。
- 已拥有装备被排除。
- 阶段匹配影响排序。
- 多名敌人的重复反制被合并。
- 输出数量和字符数受限。
- 缺失文件、损坏 JSON、未知英雄均安全降级。
- `PromptBuilder` 将机制参考放在预期位置。
- PyInstaller 规格和构建脚本包含本体目录。

现有提示词黄金哈希测试需要更新，因为用户消息会增加机制参考。新增断言应优先验证结构和关键内容，避免仅依赖脆弱的整段文本哈希。

## 非目标

第一版不包括：

- 向量数据库或语义嵌入检索。
- 在线更新本体。
- 根据敌方实时装备反推其构筑意图。
- 自动计算完整阵容胜率或经济差。
- 修改模型输出 JSON 字段。
- 将所有技能详情直接注入提示词。

这些能力可在本地检索质量得到验证后再迭代。
